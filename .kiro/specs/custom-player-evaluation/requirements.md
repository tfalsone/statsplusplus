# Requirements Document

## Introduction

Stats++ currently relies on the OOTP game engine's built-in OVR (Overall) and POT (Potential) ratings as primary inputs to player evaluation, FV grading, WAR projection, surplus calculation, and UI display. This creates two problems:

1. **League compatibility** — Some OOTP leagues disable or hide OVR/POT ratings, making Stats++ non-functional for those leagues. The platform should work with any league configuration.
2. **Analytical independence** — OVR/POT are opaque composite numbers generated by the game engine. Building evaluation on top of them means Stats++ inherits the engine's biases and provides no independent analytical value beyond what the game already shows.

This feature replaces OVR/POT dependency with a custom evaluation system built from individual player tool ratings (contact, power, stuff, movement, control, defensive grades, etc.), stats, and age/level context. The system produces its own composite scores that serve the same role OVR/POT currently fill — but are transparent, configurable, and available regardless of league settings.

The primary user value is **talent identification**: surfacing players the game undervalues, flagging players the game overvalues, and providing evaluation in leagues where the game provides none.

## Glossary

- **Tool_Rating**: An individual player skill rating from the OOTP engine (e.g., contact, power, stuff, movement, control, speed, eye). These are always available regardless of league configuration.
- **Composite_Score**: A Stats++-generated numeric score (on the 20-80 scouting scale) representing current ability. Computed from weighted Tool_Ratings, positional context, and (for MLB players) stat performance. Replaces OVR as the primary evaluation input.
- **Ceiling_Score**: A Stats++-generated numeric score (on the 20-80 scouting scale) representing projected peak ability. Computed from potential Tool_Ratings and development context. Replaces POT as the primary ceiling input.
- **Evaluation_Engine**: The module responsible for computing Composite_Score and Ceiling_Score from raw player data.
- **Tool_Weights**: Per-position configuration defining how individual Tool_Ratings contribute to the Composite_Score and Ceiling_Score.
- **Divergence_Flag**: A signal raised when the tool-only Composite_Score (excluding stat blending) differs significantly from the game's OVR or POT, indicating a potential hidden gem or landmine. Uses tool-only scores to avoid conflating stat overperformance with genuine evaluation disagreement.
- **Tool_Only_Score**: The Composite_Score computed purely from Tool_Ratings without any stat blending. For prospects this is identical to the Composite_Score. For MLB players this is the pre-blend score used specifically for divergence detection.
- **OVR**: The game engine's built-in Overall rating — a single number summarizing current ability relative to position peers.
- **POT**: The game engine's built-in Potential rating — a single number summarizing projected ceiling.
- **FV_Model**: The existing Future Value grading system in `fv_model.py` that currently takes OVR and POT as primary inputs.
- **WAR_Model**: The existing WAR projection system in `war_model.py` that currently maps OVR to expected WAR via `peak_war_from_ovr()`.
- **Calibration_Engine**: The existing `calibrate.py` system that derives league-specific valuation tables from historical data.
- **Scarcity_Model**: The existing system in `prospect_value.py` that uses POT to determine trade scarcity multipliers.
- **Arb_Model**: The existing arbitration salary estimation system in `arb_model.py` that uses OVR to project arb salaries.
- **League_Config**: The per-league configuration abstraction in `league_config.py`.

## Requirements

### Requirement 1: Composite Score Calculation for Hitters (Prospects)

**User Story:** As a league analyst, I want Stats++ to compute a custom overall ability score for prospect hitters from their individual tool ratings, so that I can evaluate hitter talent without relying on the game's OVR rating.

#### Acceptance Criteria

1. THE Evaluation_Engine SHALL compute a Composite_Score for each prospect hitter by applying position-specific Tool_Weights to the hitter's current Tool_Ratings (contact, gap, power, eye, avoid-K, speed) and defensive grades.
2. THE Evaluation_Engine SHALL produce Composite_Scores on the 20-80 scouting scale, rounded to the nearest integer.
3. WHEN a prospect hitter has L/R split ratings available, THE Evaluation_Engine SHALL use the weighted average of split ratings (weighted by expected plate appearance distribution) rather than the overall rating alone.
4. THE Evaluation_Engine SHALL apply positional defensive weighting consistent with the existing `DEFENSIVE_WEIGHTS` in `fv_model.py` so that a catcher's Composite_Score reflects framing, blocking, and arm contributions.
5. THE Evaluation_Engine SHALL use only Tool_Ratings (not stats) for prospect hitters, since minor league stats are unavailable from the StatsPlus API.

### Requirement 2: Composite Score Calculation for Pitchers (Prospects)

**User Story:** As a league analyst, I want Stats++ to compute a custom overall ability score for prospect pitchers from their individual tool ratings and arsenal, so that I can evaluate pitcher talent without relying on the game's OVR rating.

#### Acceptance Criteria

1. THE Evaluation_Engine SHALL compute a Composite_Score for each prospect pitcher by applying role-specific Tool_Weights to the pitcher's current Tool_Ratings (stuff, movement, control) and arsenal quality.
2. THE Evaluation_Engine SHALL factor arsenal depth (number of pitches rated 45+) and top-pitch quality into the pitcher Composite_Score.
3. THE Evaluation_Engine SHALL factor stamina into the Composite_Score for starting pitchers, applying a penalty when stamina falls below 40.
4. WHEN a prospect pitcher has L/R split ratings available (stuff splits, control splits), THE Evaluation_Engine SHALL incorporate platoon balance into the Composite_Score, penalizing severe one-sided splits.
5. THE Evaluation_Engine SHALL differentiate between SP and RP scoring, applying the existing bucketing logic from `assign_bucket()` to determine role before scoring.

### Requirement 3: Composite Score Calculation for MLB Players

**User Story:** As a league analyst, I want MLB player composite scores to incorporate actual stat performance alongside tool ratings, so that the evaluation reflects what a player is actually producing on the field.

#### Acceptance Criteria

1. WHEN an MLB player has at least one qualifying season of stats (130+ AB for hitters, 40+ IP for SP, 20+ IP for RP), THE Evaluation_Engine SHALL blend stat-derived performance signals with Tool_Ratings to produce the Composite_Score.
2. THE Evaluation_Engine SHALL weight stats more heavily for MLB players with longer track records (3+ seasons weighted more than 1 season), using a recency-weighted approach consistent with the existing `stat_peak_war()` methodology.
3. WHEN an MLB player has zero qualifying seasons (recent callup, injured), THE Evaluation_Engine SHALL fall back to a tool-rating-only Composite_Score, matching the prospect methodology.
4. THE stat-derived component SHALL use rate stats normalized to league average (OPS+ for hitters, ERA+/FIP for pitchers) rather than raw stats, so that the score is comparable across league environments.
5. THE Evaluation_Engine SHALL apply the existing young-player ratings blend logic: for players under peak age whose tools suggest more upside than stats show, the Composite_Score SHALL blend toward tools rather than anchoring entirely on limited stats.
6. THE Evaluation_Engine SHALL retain the pre-blend Tool_Only_Score (computed from Tool_Ratings alone, before stat blending) for every MLB player. THE Tool_Only_Score is used for divergence detection against OVR (Requirement 6) to ensure that hot streaks or small-sample stat noise do not create false hidden-gem or landmine flags.

### Requirement 4: Ceiling Score Calculation

**User Story:** As a league analyst, I want Stats++ to compute a custom ceiling score for all players from their potential tool ratings, so that I can evaluate player upside without relying on the game's POT rating.

#### Acceptance Criteria

1. THE Evaluation_Engine SHALL compute a Ceiling_Score for each player by applying the same positional Tool_Weights to the player's potential Tool_Ratings instead of current Tool_Ratings.
2. THE Evaluation_Engine SHALL produce Ceiling_Scores on the 20-80 scouting scale, rounded to the nearest integer.
3. THE Evaluation_Engine SHALL ensure that the Ceiling_Score is greater than or equal to the Composite_Score for every player.
4. WHEN a player has scouting accuracy rated Low (`Acc=L`), THE Evaluation_Engine SHALL widen the uncertainty range on the Ceiling_Score by applying a variance penalty.
5. THE Evaluation_Engine SHALL incorporate work ethic (`WrkEthic`) as a development modifier on the Ceiling_Score: high work ethic increases the Ceiling_Score by a small amount, low work ethic decreases it.

### Requirement 5: Tool Weight Configuration

**User Story:** As a league operator, I want tool weights to be derived from actual performance data in my league, so that the evaluation system reflects which tools genuinely predict production rather than relying on hand-tuned guesses.

#### Acceptance Criteria

1. THE Evaluation_Engine SHALL load Tool_Weights from a per-league configuration file (`tool_weights.json`) stored in `data/<league>/config/`. Each league gets its own `tool_weights.json` — tool weights are always per-league, never shared across leagues.
2. WHEN no custom Tool_Weights configuration exists for a league, THE Evaluation_Engine SHALL use built-in default weights that produce reasonable scores across standard OOTP leagues.
3. THE Evaluation_Engine SHALL support separate Tool_Weight profiles for each positional bucket (C, SS, 2B, 3B, CF, COF, 1B, SP, RP).
4. IF a Tool_Weights configuration file contains invalid or incomplete data, THEN THE Evaluation_Engine SHALL fall back to default weights and log a warning.
5. THE Calibration_Engine SHALL derive Tool_Weights independently for each league by running per-league, per-positional-bucket component-level regressions using only that league's own historical data from the `ratings` and stats tables. Each league's calibration produces its own `data/<league>/config/tool_weights.json`. This ensures that a deadball-era league produces different tool weights than a steroid-era league.
6. THE Calibration_Engine SHALL run component-level regressions rather than a single WAR regression: hitting tools (contact, gap, power, eye, avoid-K, speed) SHALL be regressed against OPS+ (offensive production), baserunning tools (speed, stealing ability `steal`, stealing aggressiveness `stl_rt`) SHALL be regressed against stolen base metrics (SB, CS, SB%), fielding tools SHALL be regressed against fielding stats (ZR, errors), and pitching tools (stuff, movement, control) SHALL be regressed against FIP (defense-independent pitching). Speed is included in the hitting regression because it contributes to batting average (infield hits, beating out grounders) and slugging (stretching doubles into triples) — the regression will determine the actual magnitude of this contribution. This decomposition avoids confounding factors where a tool appears to correlate with WAR only because of unrelated value components.
7. THE Calibration_Engine SHALL recombine component-level regression weights into a single unified weight profile per positional bucket using position-specific recombination weights that reflect the relative importance of offense, defense, baserunning, and pitching for each position (e.g., 1B is mostly offense, SS has significant defensive weight, CF has high defense weight). These recombination shares are initial estimates subject to calibration against real data.
8. THE regression-derived Tool_Weights SHALL replace the built-in defaults for any positional bucket where sufficient historical data exists (meeting the minimum regression sample size).
9. WHEN insufficient historical data exists for a positional bucket (fewer than the minimum regression sample size), THE Calibration_Engine SHALL retain the built-in default weights for that bucket.
10. THE Calibration_Engine SHALL normalize regression coefficients so that Tool_Weights for each positional bucket sum to 1.0, preserving the relative importance ratios from the regression.

### Requirement 6: Divergence Detection and Flagging

**User Story:** As a league analyst, I want the system to flag players where the custom evaluation significantly disagrees with the game's OVR/POT, so that I can identify hidden gems and landmines that provide a scouting edge.

#### Acceptance Criteria

1. WHEN a league provides OVR/POT ratings, THE Evaluation_Engine SHALL compute the divergence between the Tool_Only_Score and OVR, and between Ceiling_Score and POT, for every player. THE divergence SHALL use the tool-only score (excluding stat blending) to avoid conflating stat overperformance with genuine evaluation disagreement.
2. WHEN the Tool_Only_Score exceeds OVR by 5 or more points, THE system SHALL flag the player as a potential "hidden gem" (the custom model reads the player's tools more favorably than the game's composite does).
3. WHEN OVR exceeds the Tool_Only_Score by 5 or more points, THE system SHALL flag the player as a potential "landmine" (the game rates the player higher than the individual tool profile supports — the player may be carried by one elite tool or by a game-engine weighting quirk).
4. THE system SHALL surface divergence flags in the web UI on player pages, prospect rankings, and team roster views so that users can quickly identify evaluation disagreements.
5. THE system SHALL provide a dedicated divergence report (CLI or web) that lists the top hidden gems and top landmines across the league, sorted by divergence magnitude.
6. WHEN a league does not provide OVR/POT ratings, THE system SHALL skip divergence detection without errors.
7. THE Evaluation_Engine SHALL retain the Tool_Only_Score for all MLB players (in addition to the stat-blended Composite_Score) so that divergence detection is always based on a pure tool-vs-tool comparison, not influenced by hot streaks or small-sample stat noise.

### Requirement 7: Tool Profile Insights

**User Story:** As a league analyst, I want the system to surface tool-level insights beyond a single composite number, so that I can understand why a player scores the way they do and identify specific strengths and weaknesses.

#### Acceptance Criteria

1. THE Evaluation_Engine SHALL classify each player's tool profile into an archetype (e.g., "contact-first hitter," "power-over-hit," "stuff-over-command pitcher," "elite defender," "balanced") based on the distribution of their tool ratings relative to the composite.
2. THE Evaluation_Engine SHALL identify each player's carrying tools (tools rated 15+ points above their Composite_Score) and red-flag tools (tools rated 15+ points below their Composite_Score).
3. THE player page in the web UI SHALL display the tool profile archetype and highlight carrying tools and red-flag tools visually.
4. WHEN comparing two players (e.g., in trade evaluation), THE system SHALL surface tool-level differences rather than only comparing composite numbers.

### Requirement 8: Development Tracking with Custom Scores

**User Story:** As a league analyst, I want to track how individual tools and composite scores change over time, so that I can identify which prospects are developing, stagnating, or declining at the tool level.

#### Acceptance Criteria

1. THE system SHALL store Composite_Score and Ceiling_Score in the `ratings_history` table alongside each monthly rating snapshot.
2. THE system SHALL compute tool-level deltas between snapshots (e.g., "contact +5, power -3 since last snapshot") and surface them in the player page.
3. WHEN a prospect's Composite_Score increases by 3+ points between snapshots, THE system SHALL flag the player as a "riser" in prospect rankings.
4. WHEN a prospect's Ceiling_Score decreases by 3+ points between snapshots, THE system SHALL flag the player as having "reduced ceiling" in prospect rankings.

### Requirement 9: FV Model Migration

**User Story:** As a league analyst, I want the FV grading model to use Composite_Score and Ceiling_Score instead of OVR and POT, so that prospect evaluation works in leagues without OVR/POT.

#### Acceptance Criteria

1. THE FV_Model SHALL accept Composite_Score in place of OVR and Ceiling_Score in place of POT as inputs to `calc_fv()`.
2. THE FV_Model SHALL preserve all existing FV modifiers (defensive bonus, versatility bonus, work ethic, critical tool penalty, platoon split penalty, scouting accuracy penalty, RP positional discount).
3. THE FV_Model SHALL produce FV grades within ±5 points of the current model's output for the same player when OVR and Composite_Score are within 3 points of each other (backward compatibility validation criterion).
4. WHEN a league provides OVR/POT ratings, THE FV_Model SHALL use Composite_Score and Ceiling_Score by default but allow a configuration flag to revert to OVR/POT for comparison purposes.

### Requirement 10: WAR Projection Migration

**User Story:** As a league analyst, I want WAR projections to use Composite_Score instead of OVR, so that contract surplus and trade valuation work in leagues without OVR/POT.

#### Acceptance Criteria

1. THE WAR_Model SHALL accept Composite_Score as input to `peak_war_from_ovr()` (or its replacement function) for ratings-based WAR projection.
2. THE Calibration_Engine SHALL derive Composite_Score-to-WAR regression tables (per positional bucket) in the same manner it currently derives OVR-to-WAR tables.
3. THE WAR_Model SHALL continue to prefer stat-based WAR (`stat_peak_war()`) over ratings-based WAR when sufficient stat history exists, preserving the existing blending logic in `contract_value.py`.
4. WHEN calibrated Composite_Score-to-WAR tables are not yet available (new league, insufficient data), THE WAR_Model SHALL fall back to default tables derived from the built-in Tool_Weights.

### Requirement 11: Surplus Model Migration

**User Story:** As a league analyst, I want prospect surplus and contract surplus calculations to use Composite_Score and Ceiling_Score instead of OVR and POT, so that trade valuation works in leagues without OVR/POT.

#### Acceptance Criteria

1. THE Scarcity_Model SHALL use Ceiling_Score instead of POT for the scarcity multiplier lookup in `prospect_value.py`.
2. THE prospect surplus model SHALL use Composite_Score and Ceiling_Score for the certainty multiplier (`_certainty_mult`) and realization blend calculations.
3. THE contract surplus model in `contract_value.py` SHALL use Composite_Score instead of OVR for the development ramp projection (projecting score growth toward Ceiling_Score over years to peak).
4. THE Arb_Model SHALL use Composite_Score instead of OVR for arbitration salary projection in `arb_model.py`.
5. THE `fv_calc.py` batch pipeline SHALL compute Composite_Score and Ceiling_Score for all players before running FV and surplus calculations.

### Requirement 12: Graceful Degradation for OVR/POT Availability

**User Story:** As a league operator running a league that provides OVR/POT, I want the system to still function correctly and not lose any existing capability.

#### Acceptance Criteria

1. WHEN a league's ratings data includes OVR and POT columns with valid values, THE Evaluation_Engine SHALL still compute Composite_Score and Ceiling_Score from tool ratings (OVR/POT are not used as inputs to the custom scores).
2. WHEN a league's ratings data has missing or zero OVR/POT values for all players, THE Evaluation_Engine SHALL operate normally using only tool ratings.
3. THE system SHALL store both Composite_Score and Ceiling_Score in the database alongside any available OVR/POT values, so that users can compare the custom evaluation against the game's ratings.
4. IF the `ratings` table lacks OVR/POT columns entirely, THEN THE system SHALL handle the missing columns without errors.

### Requirement 13: Calibration Integration

**User Story:** As a league analyst, I want the calibration system to derive league-specific evaluation tables and tool weights from actual performance data, so that the custom evaluation adapts to each league's data.

#### Acceptance Criteria

1. THE Calibration_Engine SHALL run Composite_Score-to-WAR regression per positional bucket using the same methodology it currently uses for OVR-to-WAR regression.
2. THE Calibration_Engine SHALL derive FV-to-Peak-WAR tables from the Composite_Score-to-WAR regression, replacing the current OVR-based derivation.
3. THE Calibration_Engine SHALL store calibrated Composite_Score-to-WAR tables in `config/model_weights.json` under a new key, preserving backward compatibility with existing OVR-based tables during the transition.
4. WHEN insufficient data exists for Composite_Score-based calibration (fewer than the minimum regression sample size), THE Calibration_Engine SHALL fall back to default tables.
5. THE Calibration_Engine SHALL derive Tool_Weights per positional bucket using component-level regressions against domain-specific stats: hitting tools (contact, gap, power, eye, avoid-K, speed) against OPS+ (computed from batting_stats and league averages), baserunning tools (speed, stealing ability `steal`, stealing aggressiveness `stl_rt`) against stolen base metrics (SB, CS from batting_stats), fielding tools against fielding stats (ZR, errors from fielding_stats), and pitching tools against FIP (computed from pitching_stats). Each regression uses only the active league's own historical data within the standard calibration window. Tool weights are stored per-league in `data/<league>/config/tool_weights.json`.
6. THE Calibration_Engine SHALL store regression-derived Tool_Weights in the league's own `data/<league>/config/tool_weights.json`, overwriting any previous calibrated weights while preserving the file format expected by the Evaluation_Engine.
7. THE Calibration_Engine SHALL run tool weight regression as a new calibration step during the refresh pipeline, after data is loaded and before composite scores are computed.
8. WHEN the tool weight regression produces a negative coefficient for a tool, THE Calibration_Engine SHALL clamp that coefficient to zero and re-normalize the remaining weights, since a negative weight would imply that a higher tool rating predicts worse performance.
9. THE Calibration_Engine SHALL recombine component regression results into unified weight profiles using position-specific recombination weights (offense share, defense share, baserunning share) that reflect each position's value composition.

### Requirement 14: Web UI Display Updates

**User Story:** As a user browsing the web dashboard, I want to see the custom Composite_Score and Ceiling_Score in place of (or alongside) OVR/POT, so that the UI reflects the new evaluation system.

#### Acceptance Criteria

1. THE player page SHALL display Composite_Score and Ceiling_Score in the header summary where OVR/POT currently appear.
2. THE team roster tables SHALL display Composite_Score in the column currently labeled "Ovr".
3. THE prospect rankings (league page top 100, team farm tab) SHALL use Composite_Score for display and Ceiling_Score for any ceiling-related context.
4. THE player search autocomplete results SHALL include Composite_Score instead of OVR.
5. WHEN a league has OVR/POT available, THE player page SHALL display the game's OVR/POT alongside the custom scores with divergence indicators (color-coded: green for hidden gem, red for landmine, neutral for agreement).
6. THE player page header SHALL use the same color-coding tiers (grade bar styling) for Composite_Score and Ceiling_Score as currently used for OVR/POT.

### Requirement 15: Database Schema Updates

**User Story:** As a developer, I want the database schema to store custom evaluation scores, so that all downstream queries and displays can access them efficiently.

#### Acceptance Criteria

1. THE `ratings` table SHALL include `composite_score`, `ceiling_score`, and `tool_only_score` columns populated during the evaluation pipeline. The `tool_only_score` stores the pre-stat-blend score for divergence detection.
2. THE `ratings_history` table SHALL include `composite_score` and `ceiling_score` columns for development tracking over time.
3. THE `player_surplus` table SHALL reference `composite_score` instead of (or in addition to) `ovr` for the stored evaluation snapshot.
4. WHEN migrating an existing database, THE schema migration SHALL add the new columns without dropping or modifying existing OVR/POT columns.

### Requirement 16: Batch Evaluation Pipeline

**User Story:** As a league operator, I want composite scores computed automatically during the refresh pipeline, so that scores are always up to date after a data pull.

#### Acceptance Criteria

1. THE refresh pipeline SHALL compute Composite_Score and Ceiling_Score for all players after ratings are loaded and before FV/surplus calculations run.
2. THE batch evaluation SHALL complete within 10 seconds for a league with 2000+ players (performance constraint consistent with the existing fv_calc.py runtime).
3. THE batch evaluation SHALL write scores to the `ratings` table so that all downstream queries (web, CLI, analysis scripts) access them via the existing `latest_ratings` view.
4. WHEN the evaluation pipeline encounters a player with incomplete tool ratings (missing fields), THE pipeline SHALL compute a partial score using available tools and flag the player as having reduced confidence.

### Requirement 17: Testability and SOLID Design

**User Story:** As a developer, I want all new functions in the Evaluation Engine to follow SOLID principles and be independently unit-testable, so that the codebase remains maintainable and every function can be verified in isolation.

#### Acceptance Criteria

1. ALL public functions in the Evaluation Engine SHALL be pure functions with clear input/output contracts — no hidden state, no global mutable variables, no side effects beyond the explicitly documented return value.
2. ALL functions that require database access SHALL accept a connection object (or equivalent data accessor) as a parameter via dependency injection, rather than creating connections internally. This enables unit testing with mock or in-memory databases.
3. EACH public function SHALL have a single responsibility — one function does one thing. Composite operations SHALL be composed from smaller, independently testable functions.
4. ALL public functions SHALL have explicit type annotations on parameters and return values, serving as machine-readable interface contracts.
5. THE unit test suite SHALL include tests for every public function in the Evaluation Engine, covering at minimum: one happy-path case, one edge case, and one error/boundary case per function.
6. THE batch pipeline entry point (`run()`) SHALL be the only function with side effects (DB writes). All computation functions it calls SHALL be pure and independently testable.

### Requirement 18: Two-Way Player Handling

**User Story:** As a league analyst, I want the evaluation system to properly handle two-way players (players who both hit and pitch), so that their full value is captured and represented in the system.

#### Acceptance Criteria

1. WHEN a player has both hitting tool ratings (contact, gap, power, eye, avoid-K, speed) and pitching tool ratings (stuff, movement, control) with non-trivial values, THE Evaluation_Engine SHALL identify the player as a two-way player.
2. THE Evaluation_Engine SHALL compute separate Composite_Scores for each role: a hitter Composite_Score using the hitter formula and a pitcher Composite_Score using the pitcher formula.
3. THE Evaluation_Engine SHALL store the primary Composite_Score as the higher of the two role scores, and SHALL store the secondary role score in a new `secondary_composite` column.
4. THE Evaluation_Engine SHALL compute a combined value estimate that reflects the player's total contribution across both roles, using a weighted combination of the two role scores where the weight reflects the expected playing time split (e.g., a pitcher who also DHs gets primary pitcher score + a fraction of the hitter score).
5. THE Ceiling_Score for a two-way player SHALL be computed for each role separately, with the primary Ceiling_Score being the higher of the two.
6. WHEN computing FV and surplus for a two-way player, THE system SHALL use the combined value estimate rather than a single-role score, consistent with the existing `_two_way_peak_war()` logic in `war_model.py` that sums batting and pitching WAR.
7. THE web UI player page SHALL display both role scores for two-way players, clearly labeling the hitter and pitcher components.
8. THE divergence detection for two-way players SHALL compare the primary Composite_Score against OVR, since OVR reflects the game engine's combined assessment.
